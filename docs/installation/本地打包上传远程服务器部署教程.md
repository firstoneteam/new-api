# 本地打包 + 上传远程服务器部署教程（纯二进制方案）

本文档介绍在**无法在服务器上 `git pull` / 联网编译**的场景下，如何在本地把 New API 打包成一个自包含的可执行文件，上传到远程 Linux 服务器全新部署并用 systemd 托管运行。

> 适用场景：服务器无外网、无 Go / Node 环境、无 Docker，或出于安全 / 网络策略只允许上传成品文件。

***

## 方案原理

- 前端（`web/default`）通过 `bun run build` 产出 `dist`，再由 Go 的 `//go:embed` **打包进二进制**，因此单个可执行文件即可提供完整前后端。
- SQLite 使用纯 Go 驱动（`glebarez/sqlite`），编译时 `CGO_ENABLED=0`，**不依赖任何 C 库**，可从 macOS / Windows 直接交叉编译出 Linux 二进制，服务器上无需安装任何运行时。
- 数据（SQLite 库、日志）落在运行目录的 `data/` 与 `logs/` 下，升级时只替换二进制即可，数据不丢。

本文档提供两种部署形态，按需选择：

- **形态一 · 单二进制（嵌入前端）**：只跑后端二进制，它自带前端页面。最简单，见第一～十章。
- **形态二 · 前后端分离（nginx 托管静态）**：nginx 托管前端 `dist` 静态文件，只把 API 路径反代给后端。见第十三章。两种形态用的是**同一个后端二进制**，区别只在是否让 nginx 直接吐前端静态文件。

***

## 一、前置要求

### 本地打包机

| 项目 | 要求 |
| ---- | ---- |
| Go | ≥ 1.25（与 `go.mod` 保持一致） |
| Bun | ≥ 1.3（构建前端，见 `web/AGENTS.md`） |
| 操作系统 | macOS / Linux / Windows 均可（支持交叉编译） |

### 远程服务器

| 项目 | 要求 |
| ---- | ---- |
| 系统 | Linux（CentOS 7+ / Ubuntu 18.04+ / Debian 10+） |
| 架构 | 一般为 `x86_64`（amd64），ARM 服务器为 `aarch64`（arm64） |
| 配置 | 至少 1 核 2G 内存 |
| 权限 | 具备 `sudo` 以配置 systemd、放行端口 |

> ⚠️ 服务器无需安装 Go、Node、Docker，仅需能运行 ELF 可执行文件的 glibc 环境（主流发行版均满足）。

***

## 二、确认目标服务器架构

在**服务器**上执行，确定要交叉编译的目标架构：

```bash
uname -m
```

| 输出 | GOARCH |
| ---- | ------ |
| `x86_64` | `amd64` |
| `aarch64` / `arm64` | `arm64` |

后续本地编译时用到的 `GOARCH` 以此为准，`GOOS` 固定为 `linux`。

***

## 三、本地构建（打包机操作）

进入项目根目录（`new-api/`）。

### 步骤 1：设置版本号

`-X ...common.Version` 会把版本写进二进制，`VERSION` 文件默认可能为空，建议显式指定一个版本标记（例如日期或 git 短哈希）：

```bash
# 用 git 短哈希 + 日期作为版本标记（可自定义）
export APP_VERSION="$(git rev-parse --short HEAD 2>/dev/null || date +%Y%m%d)"
echo "$APP_VERSION"
```

### 步骤 2：构建前端（两套主题都必须构建）

Go 二进制通过 `//go:embed` **同时嵌入** `web/default/dist`（新版前端）与 `web/classic/dist`（旧版前端），两者缺一，`go build` 会直接报错 `pattern web/classic/dist: no matching files found`。因此**两套都必须构建**：

```bash
cd web
bun install --frozen-lockfile

# 新版前端（default）
cd default
DISABLE_ESLINT_PLUGIN='true' bun run build
cd ..

# 旧版前端（classic）—— 不可省略，否则 go build 失败
cd classic
bun run build
cd ../..
```

构建产物分别位于 `web/default/dist/` 与 `web/classic/dist/`，将在下一步一起被嵌入二进制。

> 说明：用哪套前端由**运行时**的数据库选项 `theme.frontend` 决定（`default`=新版，`classic`=旧版），不是编译期选择，也没有对应环境变量。**默认值是 `classic`（旧版）**，全新部署首次打开会看到"旧版前端即将停止维护"的提示。切换到新版的方法见第十一章。

### 步骤 3：交叉编译 Go 二进制

以 **linux/amd64** 为例：

```bash
CGO_ENABLED=0 GOOS=linux GOARCH=amd64 \
  go build \
  -ldflags "-s -w -X 'github.com/QuantumNous/new-api/common.Version=${APP_VERSION}'" \
  -o new-api-linux-amd64
```

ARM 服务器（`aarch64`）改用：

```bash
CGO_ENABLED=0 GOOS=linux GOARCH=arm64 \
  go build \
  -ldflags "-s -w -X 'github.com/QuantumNous/new-api/common.Version=${APP_VERSION}'" \
  -o new-api-linux-arm64
```

- `CGO_ENABLED=0`：产出静态二进制，不依赖服务器上的 C 库。
- `-s -w`：去除符号表与调试信息，减小体积。

### 步骤 4：校验产物

```bash
file new-api-linux-amd64
# 期望输出包含：ELF 64-bit LSB executable, x86-64 ... statically linked
```

### 步骤 5（仅前后端分离需要）：单独打包前端 dist

形态二需要把新版前端 `dist` 单独传给 nginx 托管：

```bash
tar -czf new-api-web-default-${APP_VERSION}.tar.gz -C web/default dist
```

产出 `new-api-web-default-<版本>.tar.gz`，解压后为 `dist/` 目录。

***

## 四、上传到远程服务器

### 步骤 1：在服务器上创建部署目录

```bash
# 服务器上执行
sudo mkdir -p /opt/new-api/{data,logs}
sudo chown -R "$USER":"$USER" /opt/new-api
```

- `data/`：SQLite 数据库与持久化数据。
- `logs/`：运行日志。

### 步骤 2：从本地上传二进制

```bash
# 本地打包机执行，替换 user@server 与端口
scp new-api-linux-amd64 user@server:/opt/new-api/new-api
```

数据量大或需断点续传时，推荐 `rsync`：

```bash
rsync -avz --progress new-api-linux-amd64 user@server:/opt/new-api/new-api
```

### 步骤 3：赋予可执行权限

```bash
# 服务器上执行
chmod +x /opt/new-api/new-api
/opt/new-api/new-api --version   # 验证能正常运行并打印版本
```

## 五、配置环境变量

在部署目录创建 `.env`（systemd 会加载它），至少设置会话密钥：

```bash
# 服务器上执行
cat > /opt/new-api/.env <<EOF
PORT=3000
SESSION_SECRET=$(openssl rand -hex 16)
# 使用 Redis 时必填：
# CRYPTO_SECRET=$(openssl rand -hex 16)
# SQLITE_PATH=/opt/new-api/data/one-api.db
# 使用外部 MySQL/PostgreSQL 时配置：
# SQL_DSN=user:password@tcp(127.0.0.1:3306)/newapi?parseTime=true
# REDIS_CONN_STRING=redis://localhost:6379/0
TZ=Asia/Shanghai
EOF
chmod 600 /opt/new-api/.env
```

| 变量 | 说明 | 是否必填 |
| ---- | ---- | ------ |
| `SESSION_SECRET` | 会话密钥，多机部署必须一致 | **必填** |
| `CRYPTO_SECRET` | 加密密钥，使用 Redis 时必填 | 条件必填 |
| `SQLITE_PATH` | SQLite 路径，默认 `one-api.db`（相对工作目录） | 可选 |
| `SQL_DSN` | 外部数据库连接串（不填则用 SQLite） | 可选 |
| `REDIS_CONN_STRING` | Redis 连接串 | 可选 |

> 完整变量清单见项目根目录 `.env.example`。默认使用 SQLite，数据会落在工作目录（下文 systemd 设为 `/opt/new-api`），故 SQLite 文件默认在 `/opt/new-api/one-api.db`；如需放入 `data/`，取消上面 `SQLITE_PATH` 注释。

## 六、配置 systemd 托管

创建 `/etc/systemd/system/new-api.service`：

```bash
sudo tee /etc/systemd/system/new-api.service > /dev/null <<'EOF'
[Unit]
Description=New API Service
After=network.target

[Service]
User=ubuntu
WorkingDirectory=/opt/new-api
EnvironmentFile=/opt/new-api/.env
ExecStart=/opt/new-api/new-api --port 3000 --log-dir /opt/new-api/logs
Restart=always
RestartSec=5

[Install]
WantedBy=multi-user.target
EOF
```

> 注意按实际修改 `User=`（运行账户，需对 `/opt/new-api` 有读写权限）与端口。`WorkingDirectory` 决定了 SQLite 与相对路径数据的落盘位置。

启动并设置开机自启：

```bash
sudo systemctl daemon-reload
sudo systemctl enable new-api
sudo systemctl start new-api
sudo systemctl status new-api
```

## 七、验证访问

```bash
# 服务器本机自测
curl -I http://127.0.0.1:3000
```

浏览器访问 `http://服务器IP:3000`，首次进入会引导创建管理员账户。

如需公网访问，记得：

1. 云服务器安全组放行端口；
2. 主机防火墙放行（如 `sudo ufw allow 3000` 或 `firewall-cmd --add-port=3000/tcp`）。

## 八、（可选）Nginx 反向代理 + HTTPS

若希望通过域名与 443 端口访问：

```nginx
server {
    listen 80;
    server_name your-domain.com;

    location / {
        proxy_pass http://127.0.0.1:3000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        # 流式响应（SSE）必须关闭缓冲
        proxy_buffering off;
        proxy_cache off;
        proxy_read_timeout 300s;
    }
}
```

证书建议用 `certbot` 申请。启用 HTTPS 后，若需要安全 Cookie，可在 `.env` 中设置 `SESSION_COOKIE_SECURE=true` 及 `SESSION_COOKIE_TRUSTED_URL=https://your-domain.com`。

## 九、后续升级流程

由于前端已嵌入二进制、数据独立存放，升级只需替换二进制：

```bash
# 1) 本地重新构建（重复第三章步骤）得到 new-api-linux-amd64

# 2) 上传覆盖（建议先备份旧二进制）
ssh user@server 'cp /opt/new-api/new-api /opt/new-api/new-api.bak'
scp new-api-linux-amd64 user@server:/opt/new-api/new-api

# 3) 服务器上重启
ssh user@server 'chmod +x /opt/new-api/new-api && sudo systemctl restart new-api'
```

数据库表结构由程序启动时 GORM `AutoMigrate` 自动升级，一般无需手动迁移。**升级前建议备份 `data/`（或 SQLite 文件）**。

## 十一、切换到新版前端

New API 内置两套前端，均已嵌入二进制，运行时由数据库选项 `theme.frontend` 决定：

| 值 | 前端 | 说明 |
| ---- | ---- | ---- |
| `classic` | 旧版（`web/classic`） | **默认值**，首页会提示"旧版前端即将停止维护" |
| `default` | 新版（`web/default`） | 推荐使用 |

> 注意：该选项**只能存于数据库，没有对应环境变量**。全新部署首次启动即为 `classic`，需手动切到 `default`。

### 方法一：管理后台切换（推荐）

用管理员账户登录 → 进入「系统设置」→ 找到前端 / 主题相关设置 → 选择新版前端并保存。切换后刷新页面即生效（无需重启）。

### 方法二：直接改数据库

选项以 `key='theme.frontend'` 存于 `options` 表。改完后重启服务，或等待配置同步周期生效。

- SQLite：

  ```bash
  sudo systemctl stop new-api
  sqlite3 /opt/new-api/one-api.db \
    "INSERT INTO options(\`key\`,value) VALUES('theme.frontend','default') \
     ON CONFLICT(\`key\`) DO UPDATE SET value='default';"
  sudo systemctl start new-api
  ```

- MySQL：

  ```sql
  INSERT INTO options(`key`, value) VALUES ('theme.frontend', 'default')
    ON DUPLICATE KEY UPDATE value = 'default';
  ```

- PostgreSQL：

  ```sql
  INSERT INTO options("key", value) VALUES ('theme.frontend', 'default')
    ON CONFLICT ("key") DO UPDATE SET value = 'default';
  ```

## 十二、常见问题

### Q1：服务器运行报 `no such file or directory` 但文件确实存在？

多为架构不匹配（在 amd64 机器上跑了 arm64 二进制，或反之）。用 `uname -m` 核对服务器架构，重新按对应 `GOARCH` 编译。

### Q2：提示缺少 GLIBC / 动态库？

本方案 `CGO_ENABLED=0` 产出的是静态二进制，通常不会有此问题。若出现，确认编译时确实带了 `CGO_ENABLED=0`，且没有引入依赖 CGO 的构建标签。

### Q3：前端页面 404 / 空白？

说明构建二进制时前端 `dist` 缺失导致 `//go:embed` 嵌入内容为空。请确认在 `go build` **之前**已完成 `web/default` 与 `web/classic` **两套**前端构建（见第三章步骤 2）。

### Q4：数据存在哪里？如何备份？

默认 SQLite 文件位于 `WorkingDirectory`（本文为 `/opt/new-api`）。备份：

```bash
sudo systemctl stop new-api
cp /opt/new-api/one-api.db /opt/new-api/one-api.db.$(date +%F)
sudo systemctl start new-api
```

### Q5：查看运行日志？

```bash
sudo journalctl -u new-api -f        # systemd 日志
tail -f /opt/new-api/logs/*.log      # 应用日志
```

## 十三、前后端分离部署（nginx 托管静态）

本章为**形态二**：nginx 直接托管前端 `dist` 静态文件，只把 API 路径反代给后端二进制。后端仍是同一个二进制，只是它嵌入的前端不再被使用。

目录约定（可按需调整）：

- 后端：`/opt/newapi/backend/`（含 `new-api-linux-amd64`、`.env`、`data/`、`logs/`）
- 前端：`/opt/newapi/web/dist/`

### 步骤 1：上传（本地执行）

```bash
KEY=/path/to/your_ssh_key   # 如 -i 指定的私钥
HOST=root@your-server-ip

scp -i $KEY new-api-linux-amd64 $HOST:/opt/newapi/backend/
scp -i $KEY new-api-web-default-<版本>.tar.gz $HOST:/opt/newapi/web/
```

### 步骤 2：远程解压前端并配置后端（服务器执行）

```bash
mkdir -p /opt/newapi/backend/{data,logs}
chmod +x /opt/newapi/backend/new-api-linux-amd64

cd /opt/newapi/web
tar -xzf new-api-web-default-<版本>.tar.gz   # 得到 /opt/newapi/web/dist/
ls dist/index.html

cat > /opt/newapi/backend/.env <<EOF
PORT=3000
SESSION_SECRET=$(openssl rand -hex 16)
SQLITE_PATH=/opt/newapi/backend/data/one-api.db
TZ=Asia/Shanghai
EOF
chmod 600 /opt/newapi/backend/.env
```

> 后端只监听本机 3000，对外由 nginx 负责。systemd 配置参照第六章，改 `WorkingDirectory` / `ExecStart` 到 `/opt/newapi/backend` 即可。

### 步骤 3：nginx 配置

后端注册的顶层路径前缀必须**全部**反代给后端，其余走静态文件 + SPA 回退：

```nginx
server {
    listen 80;
    server_name your-server-ip;   # 有域名换成域名

    root /opt/newapi/web/dist;
    index index.html;

    # 后端 API / 中继路径
    location ~ ^/(api|v1|v1beta|pg|mj|suno|kling|dashboard)(/|$) {
        proxy_pass http://127.0.0.1:3000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        # 流式响应（SSE）必须关闭缓冲
        proxy_buffering off;
        proxy_cache off;
        proxy_read_timeout 300s;
        client_max_body_size 100m;
    }

    location /static/ {
        expires 7d;
        access_log off;
    }

    # 前端 SPA 路由回退
    location / {
        try_files $uri $uri/ /index.html;
    }
}
```

```bash
nginx -t && systemctl reload nginx
```

### 分离部署的注意点

- **前端版本不受数据库 `theme.frontend` 影响**：nginx 直接吐 `dist/` 静态文件，即使后端库仍是默认 `classic`，页面也是你上传的那套（新版）。这是分离部署相对第十一章的优势。
- **`/:mode/mj` 边缘路由**：如 `/openai/mj/...` 这类带 mode 前缀的 MJ 中继不在上面正则内，会落到 SPA 回退。若用到该形式，在 `location` 正则中补上对应规则。
- 升级前端：重新打包 `dist` tar 上传、解压覆盖 `/opt/newapi/web/dist/` 即可，无需重启后端。
